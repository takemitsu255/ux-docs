# Web体制改善

## 品質問題の原因整理

1. __コミュニケーション不足__

    要件や仕様の意図が十分に共有できておらず、Figma 上に記載された表面的な情報だけを基準に判断してしまったことが原因と考えられます。


    疑問点や曖昧な点についての確認が不十分で、誤った前提のまま作業を進めてしまったこと、あるいは Jira や Figma, Slack でのテキストベースのコミュニケーションによって認識齟齬が生じてしまった可能性もあります。

    「聞けば早いこと」を聞かずに進行してしまい、結果的に手戻りを増やした。

2. __仕様理解不足__

    対象業務・サービスの背景理解が浅く、仕様の「正しさ」や「整合性」を判断できなかった。

    Figma の内容に誤りや矛盾があっても気づけず、指摘・改善提案が行えなかった。

    ※事前告知と申し込み開始でボタンの表示/非表示、先祖返りなど

3. __レビュー体制の不備__

    仕様レビュー・UIレビュー・コードレビューのプロセスが形骸化していた、または十分でなかった。

    複数名による "ダブルチェック" が機能していなかった。

4. __役割分担・責任範囲の曖昧さ__

    「誰が仕様を最終的に確定するのか」「誰が品質を保証するのか」が不明確だった。

    上流（仕様策定）と下流（実装）の境界が曖昧で、認識がずれたまま進行した。

5. __テスト観点の不足__

    テスト仕様が UI の見た目に偏り、業務の不整合に気づけなかった。

6. __プロジェクト管理の不足__

    進捗報告が定性的で、リスクや課題の早期共有ができていなかった。

2. __仕様理解不足__
仕様の読み合わせを行う：内部でまず読み合わせをする。疑問点や不明点がでたらクライアントとのMTGを改めて設定する。また、案件の難易度（規模）が大きいものに関しては必ず作業者含めMTGを行う
・公開の順番などの判断
・初回かならず公開日を確認する　→　公開日の管理方法（聞ける状態にまずはする）
　└ 迷ったら担当者へ連絡をする
・開発担当者とのMTGの必要性（JS実装後、必要なしになり作成が無駄になるケースがある為）
3. __レビュー体制の不備__
仕様の理解が薄く、正しく機能を理解しテストできているか？疑問を感じている
・テスト仕様書の作成タスクの追加
4. __役割分担・責任範囲の曖昧さ__
役割・責任範囲は改めて、担当者と開発担当と実装役割の認識を併せてを行う
5. __テスト観点の不足__
共通項目・ソースレベルのチェックはツールの導入を検討、ヒューマンエラーを少しでも軽減する取りくみ
6. __プロジェクト管理の不足__
リスク・ボトルネックが曖昧な部分があるので、決めていく
現状内部管理のJiraでは絞り込みなどがうまく機能しなことがたたある為、BackLogなど別のツールでの管理を検討しより進捗を明確にする
また、先方Jiraでのコメントルールを変更を検討してもらう（最新のコメントがどこに記載されているか判断しづらい為、最新コメントは下に記載してもらうなどルールを決めるなど）
